Idea

I got my idea on establishing this blog from a weird teacher, who said:

A developer without a qualified blog would not be an experienced one.

On my first hearing I didn’t accept his idea. I thought I just have enough power to remember every technical details encountered, but time proved me wrong lol.

I got myself involved into an honor(progressive) course named “Advanced Language Programming”, a course for ones who evaluate himself or herself capable of learning more in programming field. The first few weeks I behaved moderately and finished every assignments in time.

But I found my problem in face of a huge assignment, which required me to write a graphic middleware (GMW) which connects graphic codes to functional codes. The deadline was three weeks after the release. I just found myself hard to grasp some important language candies, so I was constantly working like this flow:

  1. coding as normal
  2. problem found
  3. searching for solutions on the Internet, especially some dalao‘s blogs
  4. if I didn’t solve it, I look it up in Primer C++
  5. if there remains the problem, I consult my teacher
  6. goto step 1

I was “priviledged” to reach step 5 when I faced a problem, which indicates a linking failure between my source codes and public codes, seems simple yeah? It almost crushed my mind for a whole day. The teacher then recommended me to keep a blog, aiming to write down solutions to these problems.

So the establishment was at my feet.

Building progress

I admit to have encountered several failures in how to properly use GitHub Pages.

A GitHub Repository is actually a “warehouse” to stack and maintain varied types and versions of codes rather than a mainframe or a server. It is static and uncapable of actively compiling, interpreting or running codes, and GitHub Pages was just a service built upon repositories to statically show the website you pushed into the repository.

We have to reluctantly accept the reality that GitHub Pages itself does not contain a complete dynamic website. If you don’t have external mainframes or servers to store data like user’s info and blogs, then your website based on GitHub Pages have no power to support functions that normal websites have in common. Costing a lot to purchase a server is also a bad choice for we undergraduates.

However the situation has got much better after some blogging tools based on Git came into being, famous and practical like Hexo, Jekyll, Gridea, etc. They provide a technical route to help you post blogs more easily, via which your blog pages are deployed in your local device, and Git will further help you upload them onto repositories which run GitHub Pages.

I was working on my pages via GitHub Desktop before I caught these tools a glimpse. For impulse I used Gridea to quickly build a blog, but I didn’t notice potential conflicts between it and GitHub Desktop, so unfortunately my pages corrupted and cannot be written anymore.

When I write this blog, it was one year since the accident above has happened. This year I experienced Gaokao and the first semester in Tongji Univ., and deleted all repositories on GitHub for security’s sake. But now I decide to remake all of them, to complete my target, and to some more ambitious targets.

While I was writing this blog, it uses Hexo as blogging tools and NexT as a theme.

To build a blog that resemble to mine, you probably have to download and install Git and Node.js into your computer.

Click here to get Git
Click here to get Node.js

Once you finished installation, input git --version and node -v in Linux Terminal or Windows CMD to check if it outputs the version of Git or Node.js you just installed. You should have seen an option named Git Bash here when you right click in Windows File Explorer.

Then input npm install -g hexo-cli to install Hexo Client, and input hexo -v to check if your installation was a success.

If this is the first time you use GitHub Pages to build your blog, you need to register an account on GitHub and make a public repository named [your_username].github.io.

The following steps may be suitable only for Windows users.

Once all above is finished, make a new folder in your local storage. Right click in the folder, choose the Git Bash here option, and you should have gained access to a Git Command Prompt which quite resembles to Linux Terminal.

The picture following displayed the Git Command Prompt Interface. In my personal case, I built my blog in the folder F:\blog.

After you see this interface, input hexo init to make files essential for the blog, and then input npm install to install additional packages. Notice: all input operations below are in this interface, but NOT in your OS’s terminal.

If all steps above go smoothly, now you can input hexo g to generate pages and hexo s to run pages local. Open the browser and go to localhost:4000 to have a preview of your pages. You can stop running pages by pressing Ctrl-C.

Now you have a blog site that can run local. In this part you’ll learn how to write blogs and upload your sites onto the GitHub Pages repository.

All blogs you write are Markdown files and stored in the inferior folder source/_posts. You can input the command hexo new "MyBlog" to create an empty Markdown file named "MyBlog.md" in the _posts folder. To write a blog, you have to briefly grasp some fundamental rules of Markdown in tutorials. Here is a very easy example:

Effects:

H3-Title

H4-Title

I open at the close.

Name SerialNo
Zhang Rabbit CHN-001
Sam Eagle USA-001
John Bull GBT-001
Gallic Rooster FRA-001
Boris Bear RUS-001

When you finished one blog, it’s recommended that you use hexo g to generate your blogs into HTML format. This is not compulsory, but to generate is the best choice. Whether you haven’t generated your local pages since your last modification or not, you can always preview your local pages via hexo s.

This part I’ll share how to upload your local pages to your repository.

When you’re pretty sure that your GitHub account and GitHub pages repository are both ready, work on these steps as follows:

Find the global setting file _config.yml in your blog folder (In my case it’s F:\blog\_config.yml) and open it. Check up at the tail of the file to find the deploy option, and modify it like this (Remember to change my username CrackLewis to yours):

Save the file and input hexo d in the Git Bash, then Git will start uploading your pages onto your GitHub repository. From my personal experiment, the operation has a VERY HIGH rate to fail for various reasons:

  • Connect timeout
  • Auth failure
  • Blog setting failure
  • Git inner errors (rare)
  • GitHub server errors (rare)

If you’re a user from Chinese mainland, your connection fails but the network condition is quite good, it’s recommended that you get connected to a VPN to get access to GitHub.

Try to visit via https://[your_username].github.io if the deployment is a success.
